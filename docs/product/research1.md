## Real‑world “git archaeology” pain

Developers explicitly talk about “code archaeology” / “git archaeology” as a recurring, time‑consuming activity when working in large, old repos.  Blog posts describe situations where a simple `git blame` surfaces only cosmetic refactors, renames, or bulk style changes, forcing engineers to peel back many layers of history just to find the *real* commit that introduced a behavior. [foojay](https://foojay.io/today/git-archeology/)

In one “Git Archeology” article, the author describes repeatedly ending up at refactoring or formatting commits when trying to understand why a check exists, and having to use increasingly arcane blame flags and manual digging to get past the noise.  Another code archaeology guide notes that on non‑trivial projects the first blame is “at first glance” useless due to such incidental changes, so developers must manually filter the history to unearth the important decisions. [jfire](https://jfire.io/blog/2012/03/07/code-archaeology-with-git/)

There’s even a recent Reddit post from a SaaS founder measuring that answering “who made this change and why?” costs their client an average of 1.4 developer hours per incident, and pitching a product specifically to “skip git archaeology on every bug.”  That’s almost exactly your value proposition, and it shows people are willing to pay to make this go away. [reddit](https://www.reddit.com/r/SaaS/comments/1li1ia6/would_your_saas_pay_a_few_bucks_a_month_to_skip/)

## Missing “why” in legacy code

A recurring complaint in legacy code posts is that the *intent* behind code is missing, and git is the last resort to reconstruct it.  One article frames legacy code as “often not clear why it operates the way that it does,” with original devs gone, documentation nonexistent, and remaining people unable to recall details; the proposed workaround is to use `git blame` and commit messages to infer the reasoning behind arbitrary choices (e.g., “why 42 bottles of cider?”). [softwareseni](https://www.softwareseni.com/how-to-read-legacy-code-like-an-archaeologist-systematic-approaches-to-understanding-old-systems/)

The UnderstandLegacyCode blog notes that when you hit a lack‑of‑context wall, tracing back to the original decision behind the code “feels like archaeology, and it’s not easy.”  Another “read legacy code like an archaeologist” guide describes inheriting an undocumented system where developers randomly click through files or run `git blame` in frustration only to discover they themselves wrote the confusing code months earlier, underscoring how fragile human memory is for “why” decisions. [understandlegacycode](https://understandlegacycode.com/blog/7-techniques-to-regain-control-of-legacy/)

Guides on “keeping a clean git history” explicitly tell authors to think about “brave explorers” a year from now who will dig through history to figure out what was done and *why*, warning that giant squashed diffs and noisy histories make serious code archaeology “very hard to understand” and may require “extreme git mastery” to recover what happened.  That’s exactly the class of repo where an automated, history‑aware agent would be valuable. [mainmatter](https://mainmatter.com/blog/2021/05/26/keeping-a-clean-git-history/)

## Large repos, onboarding fear, and getting lost

Engineers on Q&A sites describe strong anxiety when asked to modify a repository that is “totally new” to them, saying they get lost reading file by file, that documentation is missing or outdated, and they end up repeatedly pinging a repo owner for tribal knowledge.  A senior engineer answering that question points out that at Microsoft Windows was ~50M LOC and it’s impossible to understand even 5% of the codebase, so everyone operates with limited knowledge and needs good strategies for discovering just the relevant parts. [jointaro](https://www.jointaro.com/question/KoxB3UU2zWC9pdYU6Rx0/what-is-the-effective-way-to-understand-new-repository-in-order-to-make-the-required-changes-in-that-repo/)

Multiple legacy‑code posts define a legacy codebase as one that is “hard to maintain” because of high complexity, tangled dependencies, and years of change, and recommend using metrics like “churn vs complexity” and version‑control history to locate hotspots before refactoring.  Other legacy‑code tips emphasize that version control (git or similar) is the primary “superpower” for working with old code, but also note that messy histories (e.g., many “WIP” commits squashed later) make the commit log much less useful for understanding how architecture evolved. [xebia](https://xebia.com/blog/using-metrics-to-find-the-pain-points-in-a-legacy-codebase/)

This population—engineers newly dropped into a giant monolith or multi‑repo estate and forced to understand “enough history to make a safe change”—is a strong candidate for your first customers. They already rely on git history but complain about the cost of doing it manually. [moxio](https://moxio.com/blog/ignoring-bulk-change-commits-with-git-blame/)

## AI context windows vs real monorepos

Several AI/engineering blogs explicitly call out the mismatch between LLM context windows and real‑world codebases.  One “Context Window Problem” article notes that even with very large context models (≈1M tokens), a typical enterprise monorepo can span thousands of files and several million tokens, plus documentation, tickets, and other artifacts that are also relevant. [kinde](https://kinde.com/learn/ai-for-software-engineering/best-practice/ai-context-windows-engineering-around-token-limits-in-large-codebases/)

Those same posts describe how human developers don’t load the whole repo; they iteratively traverse small sets of relevant files, following imports, references, and call graphs to build understanding—exactly the kind of multi‑hop, history‑aware exploration your agent could emulate while keeping within token budgets.  Vendor guides on “AI context windows” and “engineering around token limits” push patterns like vector search plus selective retrieval—pulling only the most relevant code chunks into the prompt rather than “fitting everything in context.” [dev](https://dev.to/akshaygupta1996/context-engineering-giving-ai-agents-memory-without-breaking-the-token-budget-1ho5)

General LLM primers explain that context windows are a hard bound on how much text the model can process at once and that complex artifacts like code or logs consume tokens quickly, making naive “stuff the repo into the prompt” approaches fragile and expensive.  That’s exactly where specialized tooling for scoped history exploration, diff‑driven retrieval, and compression of git history into explanations becomes compelling. [reintech](https://reintech.io/blog/llm-token-limits-managing-context-windows)

## Where to look for your first 100 users

Based on how and where these pains are described, your best early adopters are likely:

- Staff/principal engineers and tech leads in orgs with large monoliths or multi‑repo systems, who are responsible for safe refactors and incident response and already practice “code archaeology.” [understandlegacycode](https://understandlegacycode.com/blog/7-techniques-to-regain-control-of-legacy/)
- Devs working on legacy SaaS products, who explicitly complain about time spent answering “who made this change and why?” and have budgets for productivity tooling. [moxio](https://moxio.com/blog/ignoring-bulk-change-commits-with-git-blame/)
- Consultants and internal platform teams focused on legacy modernization, who rely heavily on git history, churn metrics, and blame to identify hotspots and understand risk. [esveo](https://www.esveo.com/en/blog/tips-for-working-with-legacy-code/)

Concretely, the language they use in public is a good guide for outreach and positioning: phrases like “git/code archaeology,” “why does this code do what it does,” “who made this change and why,” “understanding a huge new repo,” and “context window vs monorepo” show up across these sources.
